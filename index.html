03Inbound mode & options
SOCKS + HTTP proxy
Local proxy on 127.0.0.1. Point apps/browser at it manually, or via a system proxy setting.
TUN (system-wide)
Native xray-core TUN inbound — captures all device traffic. Needs elevated/admin privileges to run.
TUN mode needs to run xray-core with elevated privileges (admin/root) since it creates a system network interface. On a fresh box, also double check your OS lets the process create wintun/utun/TUN devices.
Windows specifically has several documented rough edges with xray-core's own TUN inbound (this isn't generic caution — these are real, currently open issues): wintun.dll isn't bundled and must be placed next to xray.exe manually or TUN silently fails to start; there's a currently-open bug (XTLS/Xray-core#5599) where connections stall above roughly 10–15 Mbps due to a gVisor/Windows interaction, with SOCKS mode unaffected; autoOutboundsInterface has been reported to not reliably prevent routing loops on Windows, in which case set the sendThrough field above to your physical adapter's IP as a manual fallback; IPv6 needs the adapter to already have a routable (non-link-local) address; and Windows doesn't classify xray's TUN adapter as a "Remote Access" interface, so the common built-in-Firewall killswitch trick doesn't apply to it. If TUN gives you trouble on Windows, SOCKS mode paired with a separate tool like tun2socks, or another TUN implementation (sing-box/mihomo) in front of xray, are both more mature-tested paths — several xray-core maintainers/contributors have said as much themselves.

Traffic sniffing (inbound)
Inbound access & authentication
Heads up: listening on 0.0.0.0 exposes this proxy to your whole LAN (and any network you join) with no password. Turn on the username/password requirement above before using this outside a trusted localhost-only setup.
TLS / REALITY security
Overrides the fingerprint field on both tlsSettings and realitySettings for every server, regardless of what each individual link specified. Useful for making a batch of servers behave consistently, or matching a specific browser's TLS ClientHello shape.
Cert pinning replaces normal certificate-chain trust with an exact match against this hash — get it with xray tls hash --cert <cert.pem>, xray tls ping <host:port>, or openssl x509 -noout -fingerprint -sha256 -in cert.pem. Stronger than allowInsecure since it still refuses anything that isn't this exact certificate, rather than accepting anything.
ECH hides the SNI from network-level observers (deeper than REALITY's camouflage, which still leaks the disguise domain in plaintext at the TCP/TLS layer for some analysis methods). Only applies to plain TLS security, not REALITY — REALITY already has its own SNI-camouflage mechanism.
Mux (connection multiplexing)
Heads up: this server uses XTLS Vision flow, which is generally recommended to run without Mux (Mux breaks Vision's direct TCP splicing). Consider leaving this off unless you know you need it.
